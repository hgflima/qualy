# ADR 0005 — Report visual: servidor efêmero `127.0.0.1` + export self-contained

- Status: aceito
- Data: 2026-05-04
- Relacionados: ADR 0006 (CLI determinístico com harness fino), ADR 0007 (runtime TS via `--experimental-strip-types`; carve-out do `esbuild`), ADR 0009 (distribuição via `install.sh`), `cli/src/report/server.ts`, `cli/src/report/export.ts`, `cli/src/report/data-loader.ts`, `cli/src/report/app.ts`, `cli/src/commands/report/serve.ts`, `cli/src/commands/report/export.ts`, `commands/lint/report.md`, SPEC §4 (Report visual, linhas 318–326), SPEC §6 Never (linhas 421 — loopback only; 422 — sem dados sensíveis no export), SPEC §7.7 (acceptance — server + export idênticos), `docs/report-design.md`, `docs/audit-format.md`

## Contexto

O SPEC §4 descreve um report visual com duas formas de entrega aparentemente distintas:

> Stack: TypeScript puro + Vite-like dev server **embutido** (`server.ts` usa Node `http` + esbuild via `import('esbuild')`). [...] Export: `export.ts` produz **HTML self-contained** (CSS inline, JS inline, dados JSON inline) — funciona offline, abrível direto no browser, versionável em `quality-report/<timestamp>.html`.

E o SPEC §7.7 fixa o invariante de paridade:

> No fim, oferece exportar snapshot self-contained para `quality-report/<timestamp>.html`. O HTML exportado abre offline e renderiza idêntico ao servidor.

E o SPEC §6 Never coloca duas travas duras:

> Nunca expor o servidor do report fora de `localhost` (sem `0.0.0.0`, sem túnel). Nunca embutir dados sensíveis no export do report (filtrar `process.env`, paths absolutos do filesystem do autor, tokens em config).

A pergunta que o ADR responde: **qual a fronteira certa entre o servidor live e o export estático, e por que o report é entregue por dois canais em vez de um?**

Há um espectro plausível:

- **Apenas servidor**: `/lint:report` sobe HTTP em `127.0.0.1`; share do dashboard exige tunelar (proibido pelo SPEC §6 line 421) ou screenshot manual. Versionar tendência exige imprimir o JSON ou recortar prints — perde-se a interatividade dos charts.
- **Apenas export estático**: `/lint:report` gera `quality-report/<ts>.html` direto e abre via `file://`. Sem dev seam para iterar tema/componente — cada mudança exige re-export. Sem rota para `/app.js` (bundle) durante desenvolvimento, perde-se o ciclo de feedback rápido.
- **Servidor expondo `0.0.0.0`** (com auth opcional): habilita share remoto direto, mas viola SPEC §6 line 421 e introduz superfície de auth/HTTPS/TLS/CSP fora do escopo MVP.
- **Build pré-renderizado tipo SSR** (Vite/Astro/Next produzindo `dist/` versionado): adiciona build step + dependência de framework + diretório de artefatos no repo do CLI; viola ADR 0006 (CLI determinístico simples) e ADR 0007 (sem build step para o CLI próprio).
- **Servidor + export como caminhos independentes**: dois bundles, dois formatos de payload, dois pontos de drift entre o que o usuário vê live e o que ele compartilha como anexo.

Sinais de fundo que pesam na escolha:

- O report é **read-only** e **opt-in** (SPEC §6 Ask first line 405): o usuário decide quando rodar e quando exportar; nada é gerado sem `AskUserQuestion`. Isso elimina a pressão para minimizar fricção entre canais (não é "live é a única forma").
- O snapshot exportado é **versionável em git** (SPEC §2 line 181, §7.7 line 467): vai parar em PR review, em anexo de Slack, em diretório compartilhado. O conteúdo precisa ser autossuficiente — sem `<link>`/`<script src>` apontando para fora — e **sem dados sensíveis** (cwd absoluto do autor, tokens, `process.env`).
- O servidor live é uma **conveniência local**: usuário roda `/lint:report` numa sessão Claude Code, abre o browser, fecha. Não há cache que sobreviva ao processo. Não há sessão entre máquinas. O bind em `127.0.0.1` + porta efêmera (`port=0`, kernel escolhe) é literal — sem flag de override.
- O `esbuild` já está nas dependências (`cli/package.json`) — é a única dependência de "build" do projeto, e ADR 0007 autoriza o uso **somente dentro de `report/`** (CLI próprio roda em `--experimental-strip-types` puro). Reuso de uma só ferramenta para os dois canais (server bundle + export bundle) elimina divergência de toolchain.
- O payload `ReportData` é **estável** (`docs/audit-format.md` + `docs/report-design.md` §3): mesmo blob alimenta os dois canais via `<script id="report-data" type="application/json">`. Não há razão arquitetural para emitir formatos diferentes; a única diferença entre live e offline é **transporte** (HTTP vs. inline) e **redação** (cwd local vs. cwd redacted).
- A `chart.js` + `chartjs-chart-treemap` precisam executar **antes** do `app.js` em ambos os canais para que `globalThis.Chart` esteja populado quando os componentes montarem. Servir como UMD (não como módulo ES bundleado dentro do `app.js`) tem dois benefícios: (i) o bundle do `app.js` fica menor e mais rápido de re-bundlar durante iteração, (ii) o export tem três `<script>` blocks claros (vendor 1, vendor 2, app) — fácil de auditar visualmente em PR review.

A questão é portanto: **como entregar o mesmo report por dois canais (live em loopback + estático offline) preservando paridade visual byte-a-byte, evitar drift entre eles, travar a superfície de exposição em loopback, e impedir vazamento de dados sensíveis no canal versionável?**

## Decisão

O report é entregue por **dois canais coordenados que compartilham um único pipeline de assembly**:

1. **Servidor efêmero `report/server.ts`** — `http.createServer` bound em `SERVER_HOST = "127.0.0.1"` literal (constante exportada, lockada por teste `themes-linear-design-md.test.ts`/`report-server.test.ts`); porta default `0` (kernel escolhe livre); rotas restritas a uma whitelist exportada (`ROUTES`); bundle do `app.ts` produzido lazily na primeira requisição de `/app.js` e cacheado pelo lifetime do handle; vendor UMDs servidos como `/vendor/chart.umd.js` + `/vendor/chartjs-chart-treemap.umd.js`. Sem persistência. Encerra em `SIGINT`/`SIGTERM`.

2. **Export self-contained `report/export.ts`** — produz um único arquivo `quality-report/<safe-timestamp>.html` com **CSS inline (`<style>` blocks)**, **JS inline (`<script>` blocks)** e **dados inline (`<script id="report-data" type="application/json">`)**. Mesma ordem de carregamento (anti-flash → CSS light → CSS dark → vendor chart → vendor treemap → app bundle → data inline). Redação de dados sensíveis aplicada por default.

A integração entre os dois é uma única função pura compartilhada — `bundleApp()` em `server.ts:241`, com seam injetável para teste — usada pelos dois caminhos. Tudo o mais é função pura sobre `string`/`ReportData`: `inlineReportData`, `inlineThemeCss`, `inlineAllScripts`, `injectVendorScripts`, `redactSensitiveData`, `escapeJsonForHtml`. Side effects (FS, socket) ficam isolados em `startReportServer`/`exportReport`.

Implicações concretas, lockadas em código + testes:

1. **`SERVER_HOST = "127.0.0.1"` é literal exportado, não configurável.** `cli/src/report/server.ts:60` declara a constante; `report-server.test.ts:91` assere `expect(SERVER_HOST).toBe("127.0.0.1")`. Adicionar flag `--host` é mecanicamente impossível sem editar a fonte e quebrar o teste. SPEC §6 Never line 421 vira regra de tipo, não convenção. `0.0.0.0`/`::` rejeitados arquiteturalmente — a constante é o único parâmetro de bind aceito por `listenAsync`.

2. **Whitelist hard-coded de rotas.** `ROUTES` (`server.ts:66`) enumera os 6 caminhos válidos: `/`, `/app.js`, `/themes/linear-design-md/{light,dark}.css`, `/vendor/chart.umd.js`, `/vendor/chartjs-chart-treemap.umd.js`. `routeFor(pathname)` faz match exato + rejeição de `..` e retorna `null` para tudo o mais. `handleRequest` mapeia `null` → 404. Não há FS path computation a partir de input do usuário — o servidor jamais lê arquivo cujo caminho derive da URL. Path traversal (`../etc/passwd`) é mitigado por construção, não por sanitização.

3. **Bundle lazy + cacheado.** `getAppBundle()` (`server.ts:313`) só dispara `bundleApp` na primeira requisição de `/app.js`. Subsequentes leem da memória. Coalesce concurrent requests via `appBundlePromise`. O report é read-only — não há invalidação. Custo de startup `< 50ms` (sem build); custo de primeira renderização `~200ms` (esbuild bundle + serve).

4. **`bundleApp()` é o único ponto de compilação compartilhado.** `server.ts:241` exporta a função; `export.ts:63` re-importa via `bundleApp as defaultBundleApp` e a invoca dentro de `exportReport`. Mudança no target ESM/browser/`legalComments` afeta os dois canais simultaneamente — drift mecânico impossível. Tests passam stub `bundleAppFn` para evitar invocar `esbuild` real (ver `report-server.test.ts`/`report-export.test.ts`).

5. **`esbuild` é a única dependência de build, autorizada SOMENTE dentro de `report/`.** ADR 0007 fixa que o CLI próprio roda em `--experimental-strip-types` puro, sem build step. Carve-out: `report/` precisa produzir um bundle que o browser entenda (esbuild faz tree-shake + bundling de dependências de `chart.js`/treemap dinamicamente importadas). `esbuild` em `cli/package.json` está reservado para isso. Outros subcomandos do CLI nunca o invocam (verificável por `grep -l "from \"esbuild\"" cli/src/`).

6. **Vendor UMDs resolvidos por filesystem walk, não por `require.resolve`.** `findInNodeModules` (`server.ts:99`) sobe `dirname` desde `REPORT_ROOT` até a raiz do FS procurando `node_modules/<pkg>/<subpath>`. Funciona com (a) npm workspaces hoisting (vendor em `node_modules/` da raiz do repo), (b) layouts pnpm/yarn flat, (c) layout `install.sh` em `~/.claude/skills/lint/cli/`. Evita `createRequire().resolve` porque pacotes modernos (`chart.js`) restringem o campo `exports` e recusam expor `./dist/chart.umd.js` mesmo quando o arquivo existe — o walk lê o disco direto, ignora a metadata `exports`. Comentário em `server.ts:96` documenta a decisão.

7. **Headers de defesa em profundidade, mesmo em loopback.** `respond` (`server.ts:437`) injeta em toda resposta: `Cache-Control: no-store` (audit pode ser regravado a qualquer hora), `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`. O último impede que uma página maliciosa rodando em outra origem (mesmo via `localhost:` diferente) embute o report num `<iframe>` e leia o conteúdo via injection. Loopback não é zona de confiança absoluta — qualquer processo na mesma máquina pode tentar acesso. SPEC §6 Never line 421 trata da exposição externa; este invariante adiciona barreira para o caso "outro processo local malicioso".

8. **Redação de dados sensíveis ligada por default — somente no export.** `assembleHtml` (`export.ts:286`) aplica `redactSensitiveData` quando `redact !== false`; `redact: true` é o default em `exportReport`. O servidor live **não redige** (o cwd é literal porque pertence ao usuário; o servidor existe para aquela sessão). A redação cobre 4 categorias: `cwd → "<redacted>"` (sempre, mesmo que não pareça absoluto); `<absolute paths> → "<redacted-path>"` (POSIX `/Users/...`/`/home/...`/`/var/folders/...` + Windows `C:\…`); `process.env.NAME → "<redacted-env>"`; tokens (`sk-…`, `ghp_…`, `xox[a-z]-…` com floor de 16 chars) → `"<redacted-token>"`. Implementação: `export.ts:96` (`ABSOLUTE_PATH_RE`), `export.ts:100` (`ENV_REF_RE`), `export.ts:108` (`TOKEN_RE`). Constantes públicas (`REDACTED_*`) para teste de exatidão. Project-relative (`src/foo.ts`, `.lint-audit/2026-…json`) preservados — são parte do valor do report. SPEC §6 Never line 422 vira invariante mecânico, não checklist.

9. **`escapeJsonForHtml` defende contra injection no inline data script.** `server.ts:134` substitui `</script` por `<\/script` e escapa `U+2028`/`U+2029`. Sem isso, um `audit.recommendations[].rationale` malicioso ou um filename com `</script>` quebraria o parser HTML e injetaria DOM arbitrário. Aplicada em ambos os canais (server e export). Defesa contra entrada não-confiável (audit é local, mas paths/rationales podem conter qualquer string).

10. **Snapshots NÃO entram em `.lint-manifest.json`.** `quality-report/<ts>.html` é artefato versionável do usuário, não estado interno do CLI. `/lint:uninstall` preserva todos os snapshots por construção (manifest é a fonte da verdade do que o uninstall remove; sem entry, sem remoção). Decisão complementar: `quality-report/` **não está** em `.gitignore` raiz por default — usuário decide se versiona; mas `recs-apply`/`safe-write` jamais o tocam.

11. **Servidor encerra antes da pergunta de export (orquestração no harness).** `commands/lint/report.md` passo 6 envia `SIGINT` ao processo `report-serve`, espera shutdown, **depois** pergunta via `AskUserQuestion` se o usuário quer exportar. Ordem fixa (serve → kill → ask → export) evita estado ambíguo se o usuário cancelar. `runReportServe` tem default `waitForExitFn` que escuta `SIGINT`/`SIGTERM`; testes injetam promise resolvida imediatamente.

12. **Snapshot timestamping reusa `toSafeTimestamp` do audit.** `export.ts:47` importa `toSafeTimestamp` de `audit.ts` — mesmo formato lexicalmente sortable (`YYYY-MM-DDTHH-mm-ss-sssZ`) usado pelo `audit-latest`. Isso permite ao usuário ordenar `quality-report/*.html` como um trail histórico paralelo ao `.lint-audit/*.json`, e simplifica integrar histograma "snapshots disponíveis" em iterações futuras.

## Consequências

**Positivas**

- **Paridade visual byte-a-byte (SPEC §7.7)**: o mesmo `assembleHtml` produz a página servida e a página exportada (modulo redação + tag `<link>` vs `<style>`/`<script src>` vs `<script>`). `bundleApp()` único garante que o bundle JS é literalmente o mesmo. CSS é inline em ambos no momento de servir (`inlineThemeCss` para export; live carrega via `<link>` mas o conteúdo do arquivo é o mesmo). Drift mecânico requer modificar `assembleHtml` e quebrar testes.
- **Defesa em profundidade contra exposição lateral**: `127.0.0.1` literal + `X-Frame-Options: DENY` + `X-Content-Type-Options: nosniff` + whitelist de rotas + sem path computation de URL. Mesmo um processo malicioso na mesma máquina não consegue framar o report ou explorar MIME confusion.
- **Offline-safe shareable**: snapshot exportado vai para PR review, anexo de Slack, diretório compartilhado, sem dependência de rede ou de `node_modules`. Renderiza idêntico em qualquer browser moderno (target `es2022`).
- **Sem dados sensíveis vazando para git** (SPEC §6 Never line 422): redação por default impede que o autor exporte e commite snapshot com `/Users/<name>/...` ou `process.env.OPENAI_API_KEY` no `rationale`. Override `--no-redact` existe para preview local de confiança, mas o harness de `/lint:report` jamais o passa.
- **Dev seam para iteração de tema**: contribuidor que adiciona tema novo (workflow de `docs/report-design.md` §9) edita CSS, recarrega o browser, vê o efeito imediato — sem rebuild step, sem watch mode. `getAppBundle()` lazy permite trocar `app.ts` durante a sessão e o próximo handle reflete a mudança.
- **Audit trail paralelo em `quality-report/`**: snapshots versionados formam histórico humano-legível e auditável (anexo de revisão de PR), complementar ao `history[]` do `ReportData` (que é resumo de `.lint-audit/*.json`).
- **Bundle único compartilhado**: alterar target ESM, plugins do `esbuild`, ou estratégia de bundling afeta ambos os canais simultaneamente — quem mexe não precisa lembrar de "também atualizar o export".
- **Custo de startup nulo**: sem `dist/` versionado, sem warm cache, sem watcher. `/lint:report` invoca `report-serve`, esbuild bundla na hora (~200ms), browser abre. Versionar `dist/` ferraria o git diff e introduziria janela onde "código diverge do bundle commitado".
- **Test-friendly**: `bundleAppFn`, `loadReportDataFn`, `readFileFn`, `writeFileFn`, `mkdirFn`, `waitForExitFn`, `pidFn` — todas as fronteiras de side effect têm seam injetável. Suite passa sem subir socket real (default em `report-server.test.ts`) e sem invocar `esbuild` real (stubbed bundle).

**Negativas / tradeoffs**

- **Dependência `esbuild`**: ADR 0006/0007 estabelece "Node 22.6+ direto, sem build step"; `esbuild` é a única exceção. Mantida porque o report **precisa** entregar JS pro browser (que não roda `--experimental-strip-types`). Mitigada por carve-out documentado (uso confinado a `report/`); reproducibilidade do bundle gated por `target: "es2022"`/`format: "esm"` versionados em `bundleApp()`.
- **Vendor walk frágil em layouts atípicos**: `findInNodeModules` assume que `chart.js`/`chartjs-chart-treemap` estão em `node_modules/<pkg>/dist/...` em algum ancestral. Se o usuário usar `pnpm` com `node-linker=isolated` extremo (sem hoisting) ou `yarn pnp`, o walk pode falhar. Mitigada pela mensagem de erro com `existsSync(candidate)` reportando o caminho intencional (não stack trace de módulo init); fallback de instalação via `install.sh` (ADR 0009) que coloca tudo em `~/.claude/skills/lint/cli/node_modules/` previsível.
- **Redação por regex tem falso-positivos potenciais**: paths como `/A/B` (2 segmentos) ou strings com `:` que parecem Windows drive em comentários técnicos podem ser redigidos. Mitigada por (i) gating por boundary `[^A-Za-z0-9._/\\-]` que evita pegar URLs `https://...` (o `//` é precedido por `/`, fora do set); (ii) floor de 2 segmentos para POSIX; (iii) `--no-redact` escape hatch; (iv) testes específicos em `report-export.test.ts` cobrindo dezenas de shapes.
- **Sem build step também significa: re-bundla a cada `/lint:report`**: 200ms a cada invocação. Aceito porque `/lint:report` é interativo (usuário abre o browser, navega, fecha), não loop tight. Mitigada pelo cache lazy do mesmo bundle dentro de uma sessão de serve.
- **Snapshot ≠ live em redação**: alguém debugando "por que o servidor mostra `cwd: /Users/me/...` mas o snapshot mostra `cwd: <redacted>`" precisa entender o invariante. Mitigada por (i) `redacted: boolean` no output JSON do `report-export`, (ii) `docs/report-design.md` §8 documenta a tabela de diferenças bind/bundle/dados/redação/headers/métodos.
- **`X-Frame-Options: DENY` impede iframe legítimo**: caso de uso futuro "embedar o report num dashboard interno via `<iframe src='http://127.0.0.1:port/'>`" fica bloqueado. Aceito — o caminho legítimo é (a) snapshot self-contained anexado, (b) abrir o report direto na própria origem. Cross-origin embedding mata o invariante de defense-in-depth.
- **Esbuild bundla ESM inline; navegadores antigos podem quebrar**: target `es2022` exige browser razoavelmente novo (Chrome ≥ 94, Firefox ≥ 93, Safari ≥ 15.4). Aceito — o report é local, o usuário roda em um browser que ele mesmo escolhe. Sem polyfills, sem matriz de IE.
- **`quality-report/` não versionado por default vs versionado pelo usuário**: contraste com `.lint-audit/` (gitignored por default). Decisão consciente — snapshot é artefato de revisão, audit é estado interno. Mitigada por `commands/lint/report.md` passo 9 sugerir `git add quality-report/<ts>.html` quando o export ocorre, sem auto-commit (SPEC §6 Never line 416).

## Alternativas consideradas

- **Apenas servidor (sem export)**. Rejeitada: viola SPEC §7.7 line 467–468 (oferece exportar snapshot). Compartilhar dashboard fica preso ao loopback (SPEC §6 Never line 421 proíbe `0.0.0.0`); usuário recorta screenshots ou imprime JSON manualmente — perde-se a interatividade dos charts. Audit trail em PR review fica humanamente ilegível.
- **Apenas export estático (sem servidor)**. Rejeitada: cada mudança de tema/componente exige re-export e abrir um novo `file://` URL; perde o ciclo de feedback rápido para iteração local (`docs/report-design.md` §9 de adicionar tema vira 6 passos com rebuild). Servidor live é a forma natural durante desenvolvimento; export é a forma natural para distribuição. Os dois canais não se substituem.
- **Servidor expondo `0.0.0.0` com auth opcional**. Rejeitada: viola SPEC §6 Never line 421 explicitamente. Introduz superfície de auth/token/HTTPS/CSP fora do escopo MVP. Compartilhamento legítimo é via snapshot exportado, não túnel HTTP. "Sem `0.0.0.0`, sem túnel" é regra dura, não preferência.
- **Build pré-renderizado via SSR (Vite/Astro/Next)**. Rejeitada: viola ADR 0006 (CLI determinístico simples) e ADR 0007 (sem build step para o CLI próprio). Adiciona dependência de framework + diretório de artefatos versionados em `cli/`. Caso o framework mude versão, drift entre o JS commitado e o código TS fonte vira cenário (commitamos `dist/` desatualizado?). Manter `esbuild` como única exceção dentro de `report/` é mais cirúrgico.
- **Servidor + export como caminhos independentes (dois bundles, dois pipelines)**. Rejeitada: drift inevitável. Componente ganha feature no `app.ts`; alguém esquece de propagar para o pipeline de export; snapshot diverge do live por meses até alguém perceber. SPEC §7.7 "renderiza idêntico" vira aspiracional. A decisão de compartilhar `bundleApp()` + `inlineReportData` + `escapeJsonForHtml` mecanicamente impede esse drift.
- **Daemon persistente em vez de servidor efêmero por invocação**. Rejeitada: introduz processo de longa duração, gestão de PID file, lifecycle entre sessões, IPC para "audit foi atualizado, recarrega". Viola filosofia "CLI puro, stdout JSON, exit code semântico" (ADR 0006). Custo de startup nulo (~50ms) torna efêmero a escolha óbvia.
- **Sem redação no export (usuário responsável)**. Rejeitada: viola SPEC §6 Never line 422 explicitamente. Externalizar a responsabilidade ao usuário garante leak — qualquer projeto com `process.env` em log line do `rationale` ou path absoluto em mensagem de erro vaza no primeiro export commitado. Default seguro + escape hatch (`--no-redact`) é o trade certo.
- **Redação via blocklist explícita (`process.env.OPENAI_API_KEY`, `/Users/<known>`)**. Rejeitada: blocklist exige enumerar todos os secrets, todos os usernames, todos os paths sensíveis — incompleto por natureza. Allowlist (project-relative paths permitidos; tudo o mais redigido) é incompatível com prosa do `rationale` (palavras inglesas comuns redigidas). Heurística por shape (path absoluto, env ref, token prefix) cobre o caso típico com false-positive aceitável.
- **CSP via `Content-Security-Policy` header em vez de `X-Frame-Options`**. Considerada: mais granular, mais moderno. Postergada: exige enumerar todas as origens (`script-src 'self' 'unsafe-inline'` para scripts inline; `style-src 'self' 'unsafe-inline'`; `img-src data:` para chart canvas exports). MVP usa `X-Frame-Options: DENY` por ser binário e suficiente para o invariante "não permitir embedding". Migração para CSP fica como evolução fora do escopo da v1.
- **`createRequire().resolve` para vendor UMDs**. Rejeitada: pacotes modernos (`chart.js` é o caso concreto) restringem o campo `exports` e recusam expor `./dist/chart.umd.js` mesmo com o arquivo existente em disco. `findInNodeModules` walk é mais robusto contra essa restrição e funciona em layouts hoisted/flat/install.sh (`server.ts:96` documenta o trade).

## Verificação

- **Bind locked (loopback only):** `cli/src/report/server.ts:60` declara `SERVER_HOST = "127.0.0.1"` literal; `cli/tests/unit/report-server.test.ts:91` assere `expect(SERVER_HOST).toBe("127.0.0.1")`; teste de runtime `binds to 127.0.0.1 with a kernel-assigned port` (linha 343) abre socket real e valida `handle.host === "127.0.0.1"`.
- **Whitelist de rotas + 404 + 405:** `report-server.test.ts` cobre `routeFor` para path traversal (`..`), query strings, trailing slash inválido; testes de integração validam 404 para path desconhecido, 405 para `POST`/`PUT`, 200 com body+MIME corretos para cada rota da whitelist (`ROUTES`).
- **Headers de defesa em profundidade:** `report-server.test.ts:464` (`emits security headers (no-store, nosniff, frame DENY)`) assere `cache-control: no-store`, `x-content-type-options: nosniff`, `x-frame-options: DENY` em toda resposta.
- **Bundle único compartilhado:** `cli/src/report/server.ts:241` exporta `bundleApp`; `cli/src/report/export.ts:63` importa o mesmo símbolo via `bundleApp as defaultBundleApp`.
